1.1 信息就是位+上下文

1.1.1 8位为一组的由来

hello程序的生命周期是从一个源程序(或者说源文件)开始的，即程序员通过编辑器创建并保存的文本文件，文件名是 hello.c。源程序实际上就是一个由值 0 和 1 组成的位(又称为比特)序列，8 个位被组织成一组，称为字节。每个字节表示程序中的某些文本字符。

知识推理 8位为一组的由来：

根据冯诺依曼体系：计算机一定是用二进制表示，

一个计算机系统需要有输入，输出，控制器，运算器，存储器。

人类只知道英文和十进制（老师说的）

需要把英文和十进制转换成0 1 代码

英文转换成0 1代码涉及编码集

十进制转换成0 1 涉及二进制转换

中文以一个字为一个基本单元

同样计算机中 以8位为一个基本单元 即 一个字节

1.1.2 编码集由来

编码集名称：ASCII

推理：编码集的由来

美国人发明计算机 英语26个字母

基本单元：字节 8位

2^8 = 256 >> 26 8位完全足够表示所有英文 包括 + - 等

所以引入ASCII码 有了编码就知道了 比如 a用97表示 即编码

即告诉计算机 把97变为二进制数字 则可用表示a

这样就建立了人和计算机沟通的方式

上面hello.c的代码使用ASCII表示：单字符编码 一个字符等于一个字节

这些二进制0101。。。的排列怎么在显示器显示的 就是系统每一个字符读出来（比如#） 按照编码集 查询码表 找到对应的字符 # 则驱动程序在屏幕上画出#

以上的书本解释：

hello.c 程序是以字节序列的方式储存在文件中的。每个字节都有一个整数值，对应

于某些字符。例如，第一个字节的整数值是 35, 它对应的就是字符“# ”。第二个字节舍

整数值为 105, 它对应的字符是 ’ i ‘ ，依此类推。注意，每个文本行都是以一个看不见的

换行符 ‘W 来结束的，它所对应的整数值为 10。像 hello.c 这样只由 ASCII 字符构成

的文件称为文本文件，所有其他文件都称为二进制文件。

1.1.3 上下文与基本思想

hello.c 的表示方法说明了一个基本思想：系统中所有的信息——-包括磁盘文件、内

存中的程序、内存中存放的用户数据以及网络上传送的数据，都是由一串比特表示的。

区分不同数据对象的唯一方法是我们读到这些数据对象时的上下文。比如，在不同的上下文中，一个同样的字节序列可能表示一个整数、浮点数、字符串或者机器指令。（即编码，有了编码 按照规定读出的就是正确的 如果编码是两个字符的 那么读出来又是另外的东西）再比如.c文件按一字符来读 如果是.exe文件 就按照机器指令读 但不对 exe文件有校验码

比如一个文件 改变文件扩展名时 会提示改变文件扩展名可能会导致文件不可用 可能不可用的原因就是上下文 不知道用什么编码方式去读 改了之后 在不同的上下文 按照不同的编码去读取/解释 就可能不能用了

作为程序员，我们需要了解数字的机器表示方式，因为它们与实际的整数和实数是不

同的。它们是对真值的有限近似值，有时候会有意想不到的行为表现。这方面的基本原理

将在第 2 章中详细描述。

1.1.4 旁注 C编程语言的起源：

一开始出来的是汇编代码 汇编代码最开始是打孔机 冯诺依曼说计算机里面是0 1 使用0 1写

打孔机的语言是0 1 0 1 。。。 即机器语言

机器语言不好学 不友好 将机器语言变成英文+助记符(mov add …等) 比如mov eax 1 即把1放进寄存器eax 这就是汇编语言 为了代替机器语言对人类的不友好

制定一个编码 比如前4位是操作码代表mov 后面多少位代表寄存器（eax），数（1） 制定规范

汇编再往上：

把汇编代码变成英文编译 引入了。。。中间有汇编器等等

第一代人用机器语言写了汇编器

用汇编写C语言编译器 （早期的C语言编译器就采取了一个取巧的办法：先用汇编语言编写一个C语言的一个子集的编译器，再通过这个子集去递推完成完整的C语言编译器。）

这个上下文即计算机系统 先有鸡后有蛋

C语言指针 需要先有体系结构来做铺垫

C语言是为了设计Unix操作系统的 发现能写出程序 发展了起来

C – > C++ –>JVM/Java /python

C 语言是系统级编程的首选，同时它也非常适用 于应用级程序的编写。然而，它也

并非适用于所有的程序员和所有的情况。C 语言的指针是造成程序员困惑和程序错误的

一个常见原因。同时，C 语言还缺乏对非常有用的抽象的显式支持，例如类、对象和异

常。像 C++ 和 Java 这样针对应 用级程序的新程序语言解决了这些问题。

可能越学越蠢 需要看底层了

1.2 程序被其他程序翻译成不同的格式

1.2.1 hello.c生命周期大致介绍

hello.c输出hello world的过程 生命周期

为了在系统上运行 hello.c 程序，每条 C 语句都必须被其他程序转化为一系列的低级机器语言指令。然后这些指令按照一种称为可执行目标程序的格式打好包，并以二进制磁盘文件的形式存放起来。目标程序也称为可执行目标文件。

要变成0101…的序列

从一种0101变成另一种0101

从.c文件的上下文的0101 变成 .exe（在windows下）的上下文的0101 （在Linux是.elf）

1.2.2 具体四个阶段

在 Unix 系统上，从源文件到目标文件的转化是由编译器驱动程序完成的：

linux> gcc -o hello **hello.**c

在这里，GCC 编译器驱动程序读取源程序文件 hello.c并把它翻译成一个可执行目标文件 hello。这个翻译过程可分为四个阶段完成，如图 1-3 所示。执行这四个阶段的程序（预处理器、编译器、汇编 器和链接器）一起构成了编译系统（compilation system)

预处理阶段：通过预处理器将hello.c里的原封不动的粘到hello.i 生成hello.i 是用来把所有的宏命令全部完成

预处理器（CPP)根据以字符 #开头的命令，修改原始的 C 程序。比如hello.c 中第 1 行的**#include** < stdio.h> 命令告诉预处理器读取系统头文件stdio.h 的内容，并把它直接插入程序文本中。结果就得到了另一个 C 程序，通常是以**.i** 作为文件扩展名。

编译阶段：通过cc1编译器 把hello.i 人类识别的代码 编译成人类识别的低级代码（汇编代码）生成hello.s ，它包含一个汇编语言程序。该程序包含函数 main 的定义，如下所示：

汇编阶段：通过汇编器（as） 把hello.s编译成计算机可执行的0101。。。的代码 机器语言指令。

把这些指令打包成一种叫做可重定位目标程序？（后面会具体讲 第七章链接）（relocatable object program)的格式，并将结果保存在目标文件 hello.o中。hello.o 文件是一个二进制文件，它包含的 17 个字节是函数 main的指令编码。如果我们在文本编辑器中打开 hello.o文件，将看到一堆乱码。

重定位的意思是文件偏移 本文件需要其所依赖的链接库的文件中的函数才能继续执行（比如printf函数） 所依赖的文件的地址 不知道 将printf.o编译好的目标文件重定位到另外一个地址的过程（修正 链接器需要将printf.o这个目标文件与其他文件结合就需要修改这些位置 把一个位置修改成另一个位置） 叫重定位 这里讲解不确定清除/正确 待后续具体讲解

链接阶段： 代码中有用c的标准库 库函数在系统中（已经准备好了） 需要拿过来（这里用到了printf.o）

通过链接器将printf.o和hello.o粘在一起生成真正的可执行代码

请注意，hello 程序调用了 printf 函数，它是每个 C 编译器都提供的标准 C 库中的一个函数。printf 函数存在于一个名为 printf .o的单独的预编译好了的目标文件中，而这个文件必须以某种方式合并到我们的 hello.o程序中。链接器(Id)**就负责处理这种合并。结果就得到 **hello 文件，它是一个可执行目标文件(或者简称为可执行文件）， 可以被加载到内存中，由系统执行。

1.2.3 旁注 GUN项目：

本身是来开发操作系统的 但项目没完成linux就出来了 将Linux合并到里面了叫GUN/Linux

慈善开源项目

GNU 项目已经开发出了一个包含 Unix 操作系统的所有主要部件的环境，但对核除外，内核是由 Linux 项目独立发展而来的。

GNU 环境包括 EMACS 编辑器、GCC编译器、GDB调试器、汇编器、链接器、处理二进制文件的工具以及其他一些部件。

GCC 编译器已经发展到 支持许多不同的语言，能够为许多不同的机器生成代码。支持

的语言包括 C C++ 、Fortran Java Pascal 面向对象 C 语言（ObjectivtC)和 Ada

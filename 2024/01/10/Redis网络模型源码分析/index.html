Redis网络模型

基本概念

Redis的事件
• 文件事件：socket的可读可写事件，Redis服务器通过套接字与客户端（或者其他Redis服务器）进行连接，文件事件是服务器对套接字操作的抽象，服务器与客户端的通信会产生相应的文件事件，而服务器则通过监听并处理这些事件来完成一系列网络通信操作。
文件事件有多种，监听连接，读写数据
• 时间事件：Redis中的一些定时操作（比如serverCron函数）需要在给定时间点执行，时间事件就是这类定时操作的抽象；redis只有一个时间事件，各种定时任务都封装在其中？

Reactor

redis采用的是单线程reactor。单机压测QPS可以达到10w，因此不要小看单线程的reactor,在选型时要慎重，不是说越复杂的东西就越好，适合才是最好的。

Redis 6.0之前单线程IO Reactor模型的实现，和Redis 6.0的多线程IO Reactor模型的实现

Reactor: The reactor design pattern is an event handling pattern for handling service delivered concurrently to a service handler by one or more inputs. The service handler then demultiplexes the incoming requests and dispatches them synchronously to the associated request handlers.
由定义可以看出，Reactor模型本身是一个较为宽泛的定义，满足以下条件的都可以称为Reactor模型。
• 事件驱动模式
• 能够处理一个或多个输入
• 将事件分发个对应的handlers（回调函数）

Reactor一般组成
事件多路分发器：即seletc poll epoll；将事件源注册到分发器上，比如epoll中使用epoll_create创建epoll，使用epoll_ctl注册事件源（fd）和事件（事件类型）epoll，使用epoll_wait等待事件分发。redis即eventLoop

handle（事件源），用于识别每一个事件，在linux上是文件描述符fd 
**reactor(反应器)**，用于管理事件调度及注册删除事件等。当有激活的事件时（就绪的事件），则调用器回调函数处理，没有则运行事件主循环，在redis里就是处理函数（aeProcessEvents）及里面的各个方面

event handler（事件处理器）：即回调函数

static int aeApiPoll(aeEventLoop *eventLoop, struct timeval *tvp) {

      aeApiState *state = eventLoop->apidata;
    int retval, numevents = 0;
    // 调用epoll_wait 返回就绪事件
    retval = epoll_wait(state->epfd,state->events,eventLoop->setsize,
            tvp ? (tvp->tv_sec*1000 + tvp->tv_usec/1000) : -1);
    if (retval > 0) {
        int j;

        numevents = retval;
        for (j = 0; j < numevents; j++) {
            int mask = 0;
            struct epoll_event *e = state->events+j;
               // 确定是哪个事件
            if (e->events & EPOLLIN) mask |= AE_READABLE;
            if (e->events & EPOLLOUT) mask |= AE_WRITABLE;
            if (e->events & EPOLLERR) mask |= AE_WRITABLE;
            if (e->events & EPOLLHUP) mask |= AE_WRITABLE;
        // 放入就绪列表中
            eventLoop->fired[j].fd = e->data.fd;
            eventLoop->fired[j].mask = mask;
        }
    }
    return numevents;
}

        !ProcessingEventsWhileBlocked &&

综上可见，6.0之后是多线程Reactor，注意不是多线程 多reactor，应该是单reactor加 工作线程的模式

流程总结

客户端与服务端建立连接，socket被包装为client

客户端向服务端发送命令，服务端需要读取命令，事件分发器一直处于运行中，epoll监听读事件，

  1. epoll监听到读事件（如何监听到 就是epoll_wait底层 硬件等支撑），epoll_wait返回，将这个socket放在就绪列表

  2. 在循环中判断是什么事件，读事件则调用其回调函数 readQueryFromClient

  3. readQueryFromClient中发现当前开启IO线程处理IO事件，则将client放入server.clients_pending_read

  4. 在aeMain的循环中，每次事件驱动前的beforesleep会执行handleClientsWithPendingReadsUsingThreads，

    1. 在其中将client放入io_threads_list[target_id]（
      确定哪个线程处理），并为处理线程对应的io_threads_pending[i]设置原子量

    2. 工作线程在初始化server的时候就创建好，IO线程会死循环，当io_threads_pending[i]原子量>0时开始取io_threads_list[target_id]中的client进行处理读写事件

      1. 发现是读事件则readQueryFromClient，这里readQueryFromClient解析指令，但不执行，回到handleClientsWithPendingReadsUsingThreads（主线程）

    4. IO线程执行完，主线程执行指令
